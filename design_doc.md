# Design-Doc: LLM-агент для автоматизации корпоративных API

/ основан на шаблоне команды [Reliable ML](https://github.com/IrinaGoloshchapova/ml_system_design_doc_ru/blob/main/ML_System_Design_Doc_Template.md) /

## 1. Контекст проекта

### 1.1 Бизнес-задача

Крупные компании используют множество внутренних систем: CRM, управление проектами, базы клиентов и сотрудников. Сотрудники тратят значительное время на рутинные операции — поиск информации по проектам, заполнение отчётов о рабочем времени, управление составом команды. Часть таких запросов приходит и от внешних пользователей (гостей), которым нужно получить публичную информацию о компании.

Мы разрабатываем **LLM-агента**, способного выполнять задачи пользователей через набор корпоративных API: управление проектами, сотрудниками, клиентами и внутренней Wiki. Агент принимает запрос на естественном языке, планирует последовательность действий и исполняет их через вызовы API, соблюдая при этом правила доступа и безопасности.

Решение разрабатывается в рамках соревнования [ERC3: AI Agents in Action](https://www.timetoact-group.at/events/enterprise-rag-challenge-part-3), которое моделирует реалистичные сценарии автоматизации корпоративной среды.

**Критерии успеха:**

- Успешность выполнения задач (score на платформе ERC3) более (ну пусть будет) 75% на бенчмарке **ERC3-PROD**

### 1.2 Целевая аудитория и пользователи

**Пользователи агента:**

- **Сотрудники компании** — выполняют операции в рамках своих полномочий: смотрят свои проекты, обновляют информацию о себе, управляют командой (если являются лидом проекта)
- **Руководители** — имеют расширенный доступ к данным по всем проектам и сотрудникам
- **Гости** (неавторизованные юзеры) — могут запрашивать только публично доступную информацию, не должны получать внутренние данные

**Сценарий использования:** текстовый чат. Пользователь формулирует задачу на естественном языке, агент выполняет её автономно, возвращая результат и ссылки на источники.

**Нагрузка:** задачи выполняются последовательно в рамках одной сессии-бенчмарка. В промышленном сценарии предполагается обработка сотен запросов в день.

### 1.3 Ограничения и допущения

- Агент работает только через API, предоставленные платформой ERC3. Нет доступа к произвольным внешним сервисам
- Набор доступных операций фиксирован: операции с проектами (просмотр, поиск), операции с сотрудниками (просмотр, обновление, поиск, изменение состава команды), операции с клиентами, операции с таблицами рабочего времени, операции с Wiki организации
- Агент не должен раскрывать внутреннюю информацию гостям (зарплаты, внутренние заметки, составы команд закрытых проектов)
- Wiki компании содержит актуальные правила безопасности и бизнес-логики, которым агент должен следовать
- Риск: некорректный ответ агента (утечка данных, выполнение неправомерной операции) считается провалом задачи и снижает итоговый скор

## 2. Архитектура решения

### 2.1 Общая схема

Архитектура представляет собой цикл «планирование — исполнение», реализованный через подход [Schema-Guided Reasoning (SGR)](https://abdullin.com/schema-guided-reasoning/):

```
Пользовательский запрос (текст)
        │
        ▼
┌─────────────────────────┐
│  Извлечение правил из   │
│  Wiki → кэширование     │
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│  Проверка               │  Быстрая проверка на очевидные
│  безопасности запроса   │  нарушения доступа
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│ Выполнение задачи       │
│ в ReAct стиле:          │  
│  План -> вызов          │
│  корпоративного API ->  │
│  Анализ результата ->   │  
│  Повтор                 │
└─────────────────────────┘
        │
        ▼
  Ответ пользователю
  (текст + ссылки на сущности)
```

### 2.2 Хранилище знаний / документооборот

**Источник знаний:** внутренняя Wiki компании, доступная через ERC3 API (`list_wiki`, `load_wiki`)

Wiki содержит:

- Описание компании, локации, руководство
- Правила безопасности и разграничения доступа
- Специальные правила для AI-агента и публичного чат-бота

### 2.3 Интеграции и интерфейсы

**Внешние API:**

- **ERC3 SDK** — Python-пакет, предоставляющий типизированный доступ к бенчмаркам и корпоративным API
- **OpenAI API** — для вызовов LLM

**Доступные корпоративные эндпоинты:**

- Проекты: `projects/list/`, `projects/search/`, `project/get`, `projects/team/update`, `/projects/status/update`
- Сотрудники: `/employees/list/`, `/employees/search/`, `/employees/get/`, `/employees/update/`
- Клиенты: `/customers/list/`, `/customers/search/`, `/customers/get/`
- Таблицы рабочего времени: `/time/log`, `/time/update`, `/time/get`, `/time/search`, `/time/summary/by-project`, `/time/summary/by-employee`
- Wiki: `/wiki/list`, `/wiki/load`, `/wiki/search`, `/wiki/update`

**Мониторинг:** ERC3 SDK принимает `log_llm()` с метриками каждого LLM-вызова (модель, время, количество токенов, кэшированные токены). Платформа ERC3 предоставляет веб-интерфейс для просмотра сессий, задач и взаимодействий.

### 2.4 Инфраструктура и развертывание

- **Запуск:** локально, через `python3 main.py`
- **Зависимости:** Python 3.10+, ERC3 SDK`>= 1.1.9`
- **Вычисления:** CPU-only для логики агента. GPU не требуется — модели вызываются через облачный API OpenAI до PoenRouter'а.
- **Переменные окружения:** `OPENAI_API_KEY`, `ERC3_API_KEY`
- **Среды:** dev (erc3-dev), test (erc3-test), prod (erc3-prod)

## 3. Данные и качество знаний

Все данные находятся на стороне платформы ERC3 и доступны через API. Агент не хранит собственных датасетов. Данные представляют собой симулированную корпоративную среду: вымышленные компании с сотрудниками, проектами, клиентами, таблицами рабочего времени и внутренней Wiki.

### 3.1 Сбор и предобработка данных

**Источники:**

- Wiki компании (текстовые страницы)
- Структурированные данные (проекты, сотрудники, клиенты, таблицы рабочего времени) — доступны через API

**Предобработка:**

- Wiki страницы загружаются целиком и передаются LLM для извлечения правил
- Структурированные данные используются as-is в формате JSON (Pydantic-сериализация)
- Контекст текущего пользователя (`who_am_i`, `get_employee`) добавляется в системный промпт

### 3.2 Векторизация и индексирование

В текущей реализации **векторная база не используется**. Причины:

- Набор документов в Wiki невелик (единицы страниц) — помещается в контекстное окно LLM целиком
- Основная работа агента — вызов API, а не поиск по документам
- Правила извлекаются из Wiki один раз и кэшируются как структурированный JSON

Возможно добавим embedding-модель, БД, если проект окажется слишком простым) для выборочного включения правил в промпт.

### 3.3 Метрики качества знаний

- **Покрытие:** все правила из Wiki должны быть извлечены. Верификация — ручной просмотр JSON.
- **Консистентность:** данные в API платформы консистентны; ответственность за целостность лежит на стороне ERC3.

## 4. Модель и генерация

### 4.1 Выбор LLM и промптинг

**Модель:** OpenAI GPT-oss (бесплатный OpenRouter), мб попробуем и другие.

**Почему:** модели OpenAI поддерживают Structured Output — возможность гарантированно получить ответ, валидный относительно заданной JSON Schema / Pydantic модели. Это критично для SGR-подхода, где LLM должна вернуть не свободный текст, а конкретный вызов API с параметрами.

**Промпт-структура:**

1. Системный промпт:
  - Описание роли агента и компании
  - Извлечённые из Wiki правил безопасности (фильтруются по роли: guest / user)
  - Информация о текущем пользователе (имя, должность, отдел)
  - Остальные жанные
2. Пользовательский запрос
3. Результаты предыдущих вызовов API

**Ограничения:** максимальное кол-во вызовов API в цикле ограничено 20 итерациями.

### 4.2 Контроль качества ответов

**Метрики:**

- Основная: score на платформе ERC3 (автоматическая проверка корректности выполнения задачи)
- Включает: правильность данных, корректный outcome, наличие необходимых ссылок на сущности

**Обработка ошибок:**

- API-ошибки (`ApiException`) возвращаются в контекст агента как `"ERROR: ..."` — LLM может скорректировать план
- Проверка отсекает запросы с явным нарушением безопасности до начала цикла
- Если задачу невозможно выполнить, агент завершает с outcome `отказано` или `не поддерживается`

**Защита от hallucinations:**

- Агент оперирует только данными, полученными через API (не выдумывает ID сущностей)
- Structured Output не позволяет отклоняться от заданной схемы ответа
- Правила из Wiki явно прописаны в системном промпте

### 4.3 Обучение/дообучение

Fine-tuning и RLHF **не используются**. Агент работает с базовыми моделями OpenAI.

Улучшение качества достигается за счёт:

- Инженерии промптов (формулировка правил, структура системного сообщения)
- Проектирования SGR-схемы
- Добавления кастомных тулов
- Возможно, кэширование контекста (Wiki rules) для стабильности и экономии токенов

## 5. UX / пользовательский опыт

### 5.1 Сценарии взаимодействия

**Основные сценарии:**

- Запрос информации: «Покажи мои проекты», «Кто лидер проекта X?»
- Операции: «Залогируй 8 часов на проект Y за сегодня», «Добавь сотрудника Z в команду проекта»
- Управление: «Обнови статус проекта на архивный», «Удали страницу Wiki»

**Исключительные сценарии:**

- Запрос данных, к которым у пользователя нет доступа → `отказано`
- Запрос, не поддерживаемый API → `не поддерживается`
- Ошибка API во время исполнения → агент получает ошибку в контекст и пытается скорректировать план
- Гость пытается получить внутренние данные → `отказано`

### 5.2 Диалоговая логика

- **Single-turn:** каждая задача — один запрос, один автономный цикл исполнения. Уточняющие вопросы пользователю не задаются
- **Контекст внутри задачи:** история API-вызовов и их результатов накапливается и передаются LLM на каждом шаге
- **Между задачами:** состояние не переносится
- **Тон ответа:** деловой, лаконичный. Агент сообщает результат и прикладывает ссылки на затронутые сущности

### 5.3 Метрики UX

- **Время выполнения задачи:** зависит от количества шагов и latency OpenAI API, OpenRouter модели
- **Логирование:** каждый LLM-вызов логируется через `api.log_llm()` с метриками: модель, время, prompt/completion tokens, cached tokens. Платформа ERC3 показывает полный лог взаимодействий для каждой задачи
- **Обратная связь:** score на платформе после завершения задачи (автоматическая оценка)

## 6. Безопасность, соответствие и этика

- **Разграничение доступа:** агент получает информацию о текущем пользователе через `who_am_i()` и строит промпт с соответствующими правилами
- **Проверка:** перед основным циклом LLM оценивает, не является ли запрос очевидным нарушением
- **Фильтрация ответов:** при завершении задачи автоматически удаляются ссылки на текущего пользователя (чтобы не раскрывать ID)
- **Персональные данные:** агент работает с симулированными данными на платформе ERC3. В промышленном сценарии потребуется поработать над безопастностью
- **Прозрачность:** все вызовы LLM логируются; промпты и ответы доступны для просмотра и контроля через платформу
- **Данные не отправляются «вовне» бесконтрольно** — все запросы идут через OpenAI API. В случае чувствительных данных потребуется переход на локальную модель

## 7. План внедрения и эксплуатации

### 7.1 Этапы проекта


| Фаза | Описание                                                 | Статус     |
| ---- | -------------------------------------------------------- | ---------- |
| 0    | Исследование: изучение платформы ERC3, SGR, API          | В процессе |
| 1    | MVP: базовый агент для store-бенчмарка (`store`)         | В планах   |
| 2    | Агент для erc3-dev с полной поддержкой корпоративных API | В планах   |
| 3    | Расширение: проверка, кастомные тулы (erc3-test/prod)    | В планах   |
| 4    | Оптимизация промптов и схем для повышения score          | В планах   |


### 7.2 Поддержка и эксплуатация

- **Мониторинг:** логирование через ERC3 SDK + веб-интерфейс платформы (сессии, задачи, LLM-метрики)
- **Обновление знаний:** автоматическое при изменении Wiki
- **Версионирование:** агенты для разных бенчмарков лежат в отдельных директориях; общая кодовая база в Git

## 8. Риски и допущения


| Риск                                  | Вероятность | Влияние | Митигация                                                               |
| ------------------------------------- | ----------- | ------- | ----------------------------------------------------------------------- |
| LLM галлюцинирует ID сущностей        | Средняя     | Высокое | Structured Output ограничивает формат; агент опирается на данные из API |
| Неполное извлечение правил из Wiki    | Средняя     | Высокое | Ручная верификация кэшированных правил; использование мощной модели     |
| Изменение API платформы ERC3          | Низкая      | Высокое | Фиксация версии SDK                                                     |
| Превышение лимита шагов (20 итераций) | Низкая      | Среднее | Увеличение лимита; оптимизация промпта для сокращения шагов             |
| Задержка ответов LLM                  | Средняя     | Среднее | Переход на менее популярную модель OpenRouter                           |
| Ошибочный отказ на легитимный запрос  | Средняя     | Среднее | Изменение промпта, механизма проверки легитимности                      |


**Ключевые допущения:**

- Платформа ERC3 стабильна и API соответствует документации SDK
- Wiki компании содержит все необходимые правила безопасности
- Модели OpenAI доступны и возвращают ответы в разумное время

## 9. Бюджет и ресурсы

**Человеческие ресурсы:** проектная команда из студентов курса «Современные модели» (НИУ ВШЭ).

**Технологические ресурсы:**

- OpenAI API (GPT-oss) — бесплатно через OpenRouter
- Платформа ERC3 — бесплатный доступ по API-ключу участника
- Локальная машина для запуска агента (CPU, Python 3.10+)

**Примерные затраты:**

- Одна сессия erc3-prod (~30–50 задач): бесплатно
- Итерационная разработка (десятки запусков): много бесценных человекочасов
